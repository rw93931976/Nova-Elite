This is **not** a code bug in your server — it's almost always a Windows networking / security layer silently blocking or interfering with the bind or loopback.

### Most Likely Culprits (ranked by how often this exact symptom appears in 2025–2026 Windows 11 reports)

1. **Windows Defender Firewall silently blocking Node.js inbound/listen**  
   Even for localhost. Recent builds (especially 24H2+) frequently stop showing the "Allow app through firewall?" prompt and just block "Node.js: Server-side JavaScript" or the node.exe process.

2. **Hyper-V / WSL2 virtual networking interference** (very common on dev machines)  
   When Hyper-V is enabled (even if WSL is not actively used), it creates a virtual switch that can reroute or block loopback traffic. localhost sometimes resolves to ::1 (IPv6) and gets dropped, or the bind succeeds but no packets reach it.

3. **IPv6 localhost (::1) vs IPv4 (127.0.0.1) mismatch**  
   Node v20+ on Windows prefers IPv6 when you say `listen(PORT)` without host. Windows loopback can be quirky with IPv6-only binds.

4. **Third-party antivirus/firewall** (CrowdStrike, McAfee, Norton, etc.) or corporate policy  
   These often kill Node servers harder than Defender.

5. **Rare Node v24 Windows-specific loopback bug** (less likely, but mentioned in a few GitHub issues)

### Quick Fixes — Try in This Order

Run these in **elevated PowerShell** (right-click → Run as administrator)

#### Fix 1: Explicitly allow Node.exe through firewall (fixes ~70% of these cases)

```powershell
# Find your node.exe path (run in normal PowerShell first)
where.exe node

# Usually one of:
# C:\Program Files\nodejs\node.exe
# C:\Users\Ray\AppData\Roaming\nvm\v24.7.0\node.exe (if using nvm)

# Then create the rule (replace path with yours)
New-NetFirewallRule -DisplayName "Node.js - Allow Inbound" `
  -Direction Inbound -Action Allow -Program "C:\Program Files\nodejs\node.exe" `
  -Profile Any -Enabled True

New-NetFirewallRule -DisplayName "Node.js - Allow Outbound" `
  -Direction Outbound -Action Allow -Program "C:\Program Files\nodejs\node.exe" `
  -Profile Any -Enabled True
```

Restart your Node server and immediately test:
```powershell
curl http://127.0.0.1:3505/health
# or browser: http://127.0.0.1:3505/health
```

Also go to Windows Security → Firewall & network protection → Allow an app through firewall → look for "Node.js" and make sure both Private + Public are checked.

#### Fix 2: Force IPv4 bind in every server.listen() call

In **NovaSuperBridge.cjs** (and test_server.cjs):

Change:
```js
server.listen(PORT, () => { ... });
```

to:
```js
server.listen(PORT, '127.0.0.1', () => {
  console.log(`Listening on 127.0.0.1:${PORT}`);
});
```

- Forces IPv4 loopback (bypasses IPv6 weirdness)  
- Or use `'0.0.0.0'` if you ever want LAN access too  
- Restart server, test with `curl http://127.0.0.1:3505/health`

#### Fix 3: Disable Hyper-V / WSL temporarily (test only, requires reboot)

If above doesn't work:

```powershell
# Check if Hyper-V is on
Get-WindowsOptionalFeature -Online | Where FeatureName -like "*Hyper-V*" -or "*Microsoft-Windows-Subsystem-Linux*"

# Disable (reboot required)
Disable-WindowsOptionalFeature -Online -FeatureName Microsoft-Hyper-V-All
Disable-WindowsOptionalFeature -Online -FeatureName Microsoft-Windows-Subsystem-Linux
```

Reboot, test server again. If it suddenly works → Hyper-V was rerouting loopback. You can re-enable after, but keep IPv4 bind.

#### Fix 4: Reset Windows networking stack (nuclear but fast)

```powershell
netsh winsock reset
netsh int ip reset
ipconfig /flushdns
Restart-Computer
```

Reboot, restart Node, test.
